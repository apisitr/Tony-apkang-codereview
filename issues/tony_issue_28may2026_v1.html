AttributeError: module 'GuessingBoard.GuessingBoardOperator_GUI' has no attribute 'main_menu'

Python is telling you: "I looked inside the thing called gg, and I couldn't find anything named main_menu." But main_menu definitely exists in your code — so why can't Python find it? The answer is in what gg actually IS.

Root Cause — Module vs Class vs Instance Critical

Look at these two lines in project.py:

project.py — current code 
from GuessingBoard import GuessingBoardOperator_GUI   # line 10

# ... inside main():
gg = GuessingBoardOperator_GUI                        # line 23  ← THE PROBLEM
gg.main_menu(contents[0], ...)                        # line 24  ← CRASHES HERE

When Python sees from GuessingBoard import GuessingBoardOperator_GUI, it imports the file named GuessingBoardOperator_GUI.py — the whole file. In Python, a file is called a module.

So on line 23, gg = GuessingBoardOperator_GUI makes gg point to the module object — not the class inside it, and not any instance.

GuessingBoardOperator_GUI = <module 'GuessingBoard.GuessingBoardOperator_GUI'>
  ↳ contains: class GuessingBoardOperator_GUI
              ↳ methods: main_menu(), main_game1(), end_game() …
  ↳ contains: variable gg = instance of the class

gg (in project.py) = the MODULE itself → no main_menu here!
The key confusion: The file and the class have the SAME name — both are called GuessingBoardOperator_GUI. That makes it easy to mix them up. But they are completely different things. The file is a container; the class is a blueprint; an instance is the actual object that has main_menu.

The Concept: Module → Class → Instance

Think of it like a recipe book, a recipe, and a dish:

Python Term Real World Analogy In Your Code
Module (file) The recipe book GuessingBoardOperator_GUI.py
Class One recipe inside the book class GuessingBoardOperator_GUI:
Instance The actual dish you cooked gg = GuessingBoardOperator_GUI()

main_menu() is a method on the instance (the cooked dish). You cannot call it on the recipe book — only on the dish that was made from the recipe.

GuessingBoardOperator_GUI.py — last line 
# At the very bottom of this file, there is already an instance created:
gg = GuessingBoardOperator_GUI()    # <-- instance created here!
Clue: The module already creates an instance called gg at the bottom. That instance has main_menu(). So the object you need already exists — you just need to reach it from project.py in the correct way.

The Lesson — Python Name Lookup Rules

When Python sees from GuessingBoard import GuessingBoardOperator_GUI:

  1. It finds the folder GuessingBoard/
  2. It finds the file GuessingBoardOperator_GUI.py inside it
  3. It loads that file and gives you access to it by the name GuessingBoardOperator_GUI
  4. That name refers to the module — the whole file, as an object

To get something inside the module, you use dot notation: module.thing. This is exactly the same as how you access a class method: instance.method().

Why does this matter for the future? This confusion (module vs class vs instance) is one of the most common Python mistakes — even experienced developers hit it when code gets refactored into multiple files. Once you understand it deeply, you'll spot it instantly in any codebase.

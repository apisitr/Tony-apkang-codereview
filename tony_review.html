def load_questions(path):
    """Read the question CSV once and return all columns."""
    with open(path, 'r') as file:
        rows = list(csv.DictReader(file))   # DictReader already skips header

    if len(rows) < 10:
        raise ValueError("Need at least 10 questions in the file")

    questions = [r['question'] for r in rows]
    choices_a = [r['choice_a'] for r in rows]
    choices_b = [r['choice_b'] for r in rows]
    choices_c = [r['choice_c'] for r in rows]
    choices_d = [r['choice_d'] for r in rows]
    answers   = [r['ans'].upper() for r in rows]

    for q in questions:
        if len(q) > 45:
            raise ValueError(f"Question too long: {q}")
    for a in answers:
        if a not in {'A', 'B', 'C', 'D'}:
            raise ValueError(f"Answer must be A/B/C/D, got: {a}")

    return questions, choices_a, choices_b, choices_c, choices_d, answers

Six functions → one. One file read instead of six. And it uses real column names (r['question']) instead of fragile [0][0] indexing. Cleaner, faster, safer.

class GameState:
    def __init__(self):
        self.player_name = ""
        self.point = 0
        self.question_index = 0
        self.already_answered = False

    def record_answer(self, user_choice, correct_answer):
        if self.already_answered:
            return
        if user_choice == correct_answer:
            self.point += 1
        self.already_answered = True

    def next_question(self):
        self.question_index += 1
        self.already_answered = False

# Then pass `state` into your functions instead of using globals:
def config_display(state, game_op, display, u_ans, ans, buttons):
    state.record_answer(u_ans, ans[state.question_index])
    display.config(text=f"Point: {state.point}")

Issue 4 — Using TypeError for "not enough questions" Warning

In every get_* function you have:

Before 
if len(questions) < 10:
    raise TypeError
Why this is wrong: TypeError means "you gave me the wrong kind of thing" — like passing a string when an integer was expected. "Not enough items in a list" isn't a type problem, it's a value problem. The right exception is ValueError ("the type is fine, but the value isn't acceptable").

Also: you raise TypeError with no message, so when it triggers the user sees nothing useful. Always include a message.
Tony's Fix:
After 
if len(questions) < 10:
    raise ValueError("The question file must contain at least 10 questions")

Then in main(), your except ValueError branch catches it with a clear message. Plus your test test_get_question currently expects TypeError for the "not enough" case — that test is testing your bug, not your function.

Lessons Learned (the big ideas)

  1. DRY — Don't Repeat Yourself. If you're copy-pasting code, ask: "Can this be a loop? A parameter? A class?" Repetition is a signal, not a solution.
  2. Globals are quicksand. They feel easy at first, but the deeper you go, the harder it is to climb out. Pass state through parameters, or store it in a class. Future-you will say thank you.
  3. Pick the right exception. ValueError = wrong value, right type. TypeError = wrong type entirely. Always include a message so the user knows what happened.
  4. Duplicate function names silently overwrite each other. Python doesn't warn you. One name = one function. Always.
  5. Use csv.DictReader by column name, not by index. row['question'] is clear; list(row.values())[0][0] is a puzzle. Code should read like a sentence.
